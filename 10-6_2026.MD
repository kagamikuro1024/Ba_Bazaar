# KẾ HOẠCH SỬA UI THEO QUY CHUẨN SENIOR FE

# BA Bazaar / BA Booking CRM

## 1. Tổng quan vấn đề hiện tại

Sau khi đối chiếu app hiện tại với bộ blueprint FE, hệ thống đang thiếu hoặc chưa chuẩn ở các điểm sau:

### 1.1. Thiếu chuẩn layout dùng chung

Hiện các trang đang có cảm giác mỗi trang tự layout riêng, chưa thống nhất theo một khung chung:

* Sidebar chưa đúng vai trò global chrome.
* Một số trang thiếu main action rõ ràng ở header.
* Navigation còn trùng hoặc chưa đúng theo role.
* Notification / Action Center / Inbox còn dễ gây nhầm lẫn.
* Một số action xuất hiện sai vị trí, ví dụ action tạo request đặt trong dashboard/card thay vì global action hoặc page header.

### 1.2. Dashboard chưa chuẩn Stat Card / Filter Card

Dashboard hiện có nhiều card nhưng chưa phân biệt rõ:

* Card nào chỉ là số liệu để đọc.
* Card nào là filter một chạm cho list bên dưới.
* Card nào là action.

Theo chuẩn mới:

* **Stat Card**: chỉ hiển thị số liệu, không lọc danh sách.
* **Filter Card**: bấm vào sẽ lọc đúng list bên dưới, bắt buộc có hover/active state.
* **Action**: không đặt thành stat card; action phải nằm ở page header hoặc global action trong sidebar.

### 1.3. Action Center chưa đúng chuẩn list/table

Action Center cần là một trang xử lý request dạng list/table, không nên là dashboard thứ hai.

Hiện cần chuẩn hóa lại:

* Quick tabs.
* Search trong danh sách.
* Nút Lọc mở popover.
* Active filter chips.
* Sort đặt đúng ở header bảng.
* Row action đặt cuối mỗi dòng.
* Pagination.
* Bulk action khi chọn nhiều dòng.

### 1.4. Filter/Search chưa thống nhất

Hiện filter giữa các trang có thể đang làm khác nhau.

Cần chuẩn hóa:

* Quick tabs cho filter phổ biến.
* Search trong bảng/list.
* Advanced filter dạng popover.
* Hiển thị chip đang lọc.
* Có nút clear từng filter hoặc clear all.
* Không dùng modal lớn cho filter.

### 1.5. BA Directory và các list khác chưa tận dụng chuẩn table/list

BA Directory, My Requests, Action Center, Reports nên dùng cùng một hệ quy chuẩn:

* Header rõ.
* Filter bar thống nhất.
* Sort thống nhất.
* Row/card action rõ.
* Empty/loading/error state nhất quán.

# 2. Mục tiêu sửa đổi

Mục tiêu là đưa UI về cùng một design language để người dùng nhìn trang nào cũng hiểu ngay:

* Đang ở trang nào.
* Có bao nhiêu việc cần xử lý.
* Nút chính của trang là gì.
* Bộ lọc nằm ở đâu.
* Sort nằm ở đâu.
* Action xử lý từng dòng nằm ở đâu.
* Card nào là số liệu, card nào là filter.

Không sửa business logic booking.

Giữ nguyên:

```ts
start_date
end_date
capacity_percent
```

Không thêm booking theo giờ.

# 3. Các hạng mục cần sửa

## P0 — Cần làm ngay

### P0.1. Chuẩn hóa App Shell / Sidebar

Cần tạo hoặc chỉnh `LayoutShell` để sidebar là global chrome thống nhất.

Sidebar cần có:

* Brand / logo.
* Global search.
* Navigation theo role.
* Global action.
* User info.
* Notification bell.

Yêu cầu:

* BA Manager thấy: Dashboard, Action Center, Timeline, BA Directory, Reports.
* PM/PO thấy: My Requests, Timeline, BA Directory.
* BA thấy: My Schedule, Timeline nếu được phép read-only.
* Notification không nên là một menu lớn trùng với Action Center, mà nên là bell/icon ở sidebar hoặc header.
* Sidebar có thể collapse nếu đang chiếm nhiều chỗ.

### P0.2. Chuẩn hóa Page Header

Mỗi page phải có:

* Title chính.
* Subtitle mô tả ngắn.
* Main action ở góc phải nếu trang có action chính.

Ví dụ:

* Dashboard: main action có thể là đổi timeframe/export, không phải create request bừa bãi.
* Action Center: không cần nút tạo request, tập trung xử lý request.
* My Requests: có nút Create Request.
* BA Directory: có nút Create BA với Manager, Request BA với PM/PO ở từng row/card.

### P0.3. Chuẩn hóa Stat Card / Filter Card

Tạo 2 component riêng:

```txt
StatCard
FilterCard
```

Quy tắc:

* StatCard: chỉ xem số, không active state.
* FilterCard: bấm vào lọc list bên dưới, có hover/active state.
* Không dùng card để làm action tạo/import.

Áp dụng cho Dashboard và Action Center.

### P0.4. Sửa Action Center theo chuẩn List/Table

Action Center phải thành bảng/list compact.

Cấu trúc:

```txt
Title + Subtitle
Quick Tabs
Search trong bảng
Nút Lọc
Active Filter Chips
Bulk Action nếu chọn dòng
Table/List chính
Pagination
```

Cột đề xuất:

```txt
Priority
Type
Project
Requester
Requested / Assigned BA
Date Range
Status
Flags
Action
```

Row action:

* Review.
* Assign.
* Approve.
* Reject.
* View.

### P0.5. Chuẩn hóa Filter Component

Tạo component dùng chung:

```txt
FilterBar
AdvancedFilterPopover
ActiveFilterChips
```

FilterBar gồm:

* Quick tabs bên trái.
* Search trong list.
* Nút Lọc có badge số điều kiện.
* Popover lọc nâng cao.
* Active chips bên dưới.

Không dùng modal full-screen cho filter desktop.

### P0.6. Sửa My Requests PM/PO

My Requests nên dùng card grid:

```txt
Desktop: 3 card / hàng
Tablet: 2 card / hàng
Mobile: 1 card / hàng
```

Card phải gọn:

* Project.
* Status.
* Priority.
* Date range.
* Requested/Assigned BA.
* Capacity.
* Action.

Edit request đổi thành popup/modal.

### P0.7. Sửa BA Directory theo chuẩn table/list

BA Directory cần hiển thị ngay:

* Name.
* Level.
* Skills.
* Current capacity.
* Current project.
* Status.
* CTA.

CTA:

* Manager: View Profile, Edit, Assign nếu đang xử lý request, Create BA.
* PM/PO: Request BA.
* BA On Leave/Resigned: disable request.

### P0.8. Timeline compact + sort capacity

Timeline vẫn là màn quan trọng, nhưng cần sửa theo các chuẩn đã review:

* Week view có navigation dạng `[‹] Tuần 23 · 03/06 - 09/06 [›]`.
* Month view co theo tuần.
* Quarter view co theo tháng, dễ nhìn hơn.
* Booking bar giữ tương quan vị trí/tỉ lệ theo ngày trong timeframe.
* Highlight ngày/tuần/tháng hiện tại.
* Click cột BA sort theo capacity.

## P1 — Làm sau khi P0 ổn

### P1.1. Chuẩn hóa badge màu

Cần tạo bộ badge dùng chung:

```txt
PriorityBadge
StatusBadge
FlagBadge
CapacityBadge
```

Priority:

```txt
Low = gray
Medium = blue
High = amber/orange
Urgent = red
```

Status:

```txt
Pending = yellow
Approved/In progress = blue/green
Rejected = gray hoặc đỏ tùy rule cuối cùng đã chốt
Completed = gray
Cancelled = light gray/line-through
```

Flag:

```txt
Unassigned
Capacity Risk
Overbooked
Bench
Urgent
```

Không để flag trùng nghĩa.

### P1.2. Loading / Empty / Error State

Tạo component dùng chung:

```txt
LoadingState
EmptyState
ErrorState
AccessDenied
```

Áp dụng cho:

* Dashboard.
* Action Center.
* Timeline.
* BA Directory.
* My Requests.
* My Schedule.
* Reports.

### P1.3. Reports / Analytics theo table blueprint

Reports nên dùng cùng chuẩn list/table:

* Title/subtitle.
* Timeframe action.
* Filter/search nếu có.
* Table có sort.
* Export là page action, không nằm trong stat card.
